> This is a **worked example** of a `data.md` pillar for an adopter to reference. It illustrates a typical Drizzle + Postgres setup. Replace this content with your project's actual conventions when adopting.

## Scope

This pillar covers the project's data layer: schema definitions, migrations, query patterns, and storage decisions. It does not cover request/response contracts (those live in `api.md`) or caching (`cache.md` in Tier 2).

## Context

The project uses **Drizzle ORM** against a **Postgres 16** instance. Schema definitions live in `src/db/schema.ts`. Migrations are generated by `drizzle-kit` and stored in `src/db/migrations/`. Connection setup lives in `src/db/client.ts`.

**File layout:**

```
src/db/
├── client.ts          # postgres client + drizzle instance
├── schema.ts          # all table definitions, one file
├── migrations/        # generated migrations, never hand-edited
├── seeds/             # seed scripts, one per environment
├── queries/           # complex queries; trivial ones live with feature code
└── raw.ts             # the only place raw SQL is allowed
```

**Conventions:**

- One `schema.ts` for all tables. Splitting by domain was considered and rejected: cross-table relations are clearer in one file.
- Migration filenames follow drizzle-kit's default (`<timestamp>_<name>.sql`).
- Table names are plural snake_case (`users`, `audit_events`).
- Column names are snake_case in the DB; Drizzle maps to camelCase in TS.
- Every table has `id`, `created_at`, `updated_at`. The `updated_at` column is maintained by a trigger, not application code.

**Multi-tenancy:** every tenant-scoped table has a `tenant_id` column. Tenant isolation rules live in `data/multi-tenant.md`.

## Decisions

- **Drizzle over Prisma.** Reason: we need raw SQL escape hatches (`db/raw.ts`) for analytics queries and we wanted migration introspection that doesn't require a separate migration engine running.
- **Trigger-maintained `updated_at`.** Reason: application code forgets to update it; triggers are the only reliable source.
- **One schema file.** Reason: forces consideration of cross-table relations when adding fields.
- **`db/raw.ts` as the only raw-SQL allowance.** Reason: lets the audit logger middleware intercept queries that bypass the ORM.

## Rules

- **Never write raw SQL outside `src/db/raw.ts`.** The audit logger only intercepts queries that go through the ORM or through `raw.ts`'s wrapper. Direct `postgres.unsafe()` calls anywhere else are unaudited.
- **Never hand-edit a migration file in `src/db/migrations/`.** Regenerate via `drizzle-kit generate` from a schema change. Hand-edits desync schema.ts from the migration history.
- **Every tenant-scoped table includes `tenant_id` as the first non-id column** and a foreign-key constraint to `tenants.id`.

## Workflows

- **Adding a column:**
  1. Edit `src/db/schema.ts`.
  2. Run `pnpm db:generate` (wraps `drizzle-kit generate`).
  3. Inspect the generated migration in `src/db/migrations/`.
  4. Run `pnpm db:push` against your local DB to apply.
  5. Update relevant seeds if the column is non-nullable without a default.

- **Adding a complex query:**
  1. If used in more than one feature, add it to `src/db/queries/<domain>.ts`.
  2. If used in one place, colocate it with the feature.
  3. If it needs raw SQL, add a wrapper in `src/db/raw.ts`.

## Watchouts

- **Adding columns in dev without a migration causes drift across machines** because the prod pipeline only runs declared migrations. Always go through `drizzle-kit generate`.
- **Drizzle's `select()` without a `where` returns the whole table.** Fine in dev; catastrophic in prod for large tables. Always include `where` or `limit` when prototyping against a populated DB.
- **The `tenants` table is the root of tenant isolation.** A bug that allows cross-tenant joins is a security incident, not a feature bug. Pair with `auth.md` and `security.md` when touching anything tenant-scoped.

## Touchpoints

- `must_read_with: [auth, config]` — auth determines tenant identity; config holds DATABASE_URL and connection pool settings.
- `see_also: [api, observe]` — API endpoints often query the DB; observe.md covers query-latency metrics.

## Gaps

- **Read replica strategy** is undecided. Currently a single primary; if read load grows, we'll need rules for which queries can hit a replica.
- **Soft delete vs. hard delete** policy is inconsistent. Some tables have `deleted_at`; most don't. Needs a project-wide decision.
